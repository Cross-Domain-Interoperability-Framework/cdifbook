# Setup and Configuration Review

Date: 2026-03-11
Repository: `Cross-Domain-Interoperability-Framework/cdifbook`

## Summary for Maintainer

The repository still looks like a classic Jupyter Book project, but fresh environments now resolve a newer `jupyter-book` release that no longer behaves as this repository expects. This reproduces with both `pip` and `uv`, so the issue is not the installer choice. It is a dependency and toolchain compatibility problem.

In addition, `_config.yml` likely contains stale repository metadata for source or edit links: the default branch is configured as `master` while the current remote default branch is `main`. The `path_to_book: cdifbook` value should also be reviewed carefully. Jupyter Book defines this as the path to the book source relative to the repository root, which is different from the deployed GitHub Pages URL path `/cdifbook`.

The practical decision is whether to stabilize the current setup by pinning a compatible Jupyter Book version and correcting the metadata, or to treat this as a deliberate migration to the newer Jupyter Book / MyST toolchain. The first path is lower risk and faster. The second path is reasonable, but it should be handled as a real migration rather than a small dependency update.

## Purpose

This note summarizes the current state of the repository setup and configuration so the maintainer can decide whether to stabilize the current Jupyter Book workflow or migrate intentionally to the newer toolchain.

## Executive Summary

The repository structure is consistent with the classic Jupyter Book model built around `_config.yml` and `_toc.yml`, but a fresh install now resolves to `jupyter-book 2.1.2`, whose CLI behavior no longer matches the documented local workflow.

As a result, the current setup is not reliably reproducible for a new contributor using a fresh environment. This reproduces with both `pip` and `uv`. The root cause is the unpinned dependency specification, not the choice of installer.

There are also two smaller but important metadata drifts in `_config.yml`: the configured book path does not match the repository layout, and the configured branch does not match the default branch on the remote.

## Findings

### 1. Dependency resolution currently breaks the documented build workflow

Severity: High

`requirements.txt` is unpinned:

```text
jupyter-book
matplotlib
numpy
```

Installing from `requirements.txt` in a fresh environment currently resolves `jupyter-book 2.1.2`.

The documented build command in `README.md` is:

```bash
jupyter-book build .
```

Under that version, the command does not behave like the classic Jupyter Book workflow expected by this repository. Running the documented setup path with either `pip` or `uv` fails instead of producing an HTML book build.

Observed behavior:

- A fresh install from `requirements.txt` completed successfully with both `pip` and `uv`.
- `jupyter-book --version` reported `v2.1.2` in both cases.
- The build failed with an `EISDIR` error and did not produce the expected site build in both cases.

Impact:

- A fresh contributor following the documented setup will not get a working local build.
- The problem sits at the dependency and toolchain layer, not just in the README wording.

### 2. Repository metadata in `_config.yml` should be reviewed

Severity: Medium

The current configuration includes:

```yaml
repository:
  url: https://github.com/Cross-Domain-Interoperability-Framework/cdifbook
  path_to_book: cdifbook
  branch: master
```

Verified repository state:

- The repository root is the `cdifbook` directory itself.
- The default remote branch is `main`.
- The published site is served from `/cdifbook`, but that deployment path is not the same as Jupyter Book's `repository.path_to_book` setting.

Issues:

- `branch: master` does not match the remote default branch, which is `main`.
- `path_to_book: cdifbook` may or may not be correct, but it should not be justified by the deployed URL alone. According to Jupyter Book's configuration reference, `path_to_book` refers to the path of the book source folder relative to the repository root.

Impact:

- Source, edit, or repository links generated by the published book may point to incorrect locations or the wrong branch if this metadata does not match the repository layout.

### 3. The README currently documents a setup flow that is not reproducible

Severity: Medium

The README now offers both `pip` and `uv` installation paths plus a development workflow, but those instructions still depend on the dependency set in `requirements.txt` being compatible with the repository structure.

At the moment, the documented sequence is internally coherent, but it is not operational for a fresh environment because the resolved `jupyter-book` version is incompatible with the expected build pattern.

Impact:

- Documentation and environment state have drifted apart.
- Contributors can follow the instructions correctly and still fail.

### 4. `.gitignore` does not cover the documented local virtual environment

Severity: Medium

The README instructs contributors to create `.venv`, but `.gitignore` currently contains only:

```text
/_build
```

Impact:

- Contributors may immediately get untracked `.venv/` noise after following setup instructions.
- This is small but avoidable friction in local development.

### 5. Environment management is not yet reproducible across tools

Severity: Low

The repository currently does not contain:

- `pyproject.toml`
- `uv.lock`
- `.python-version`

This does not prevent either `pip` or `uv` from being used, but it means:

- dependency resolution is time-dependent,
- environments are not locked across contributors,
- and the project does not yet define a reproducible environment workflow.

### Build validation

The setup was tested with both installation paths documented or implied by the repository.

#### `uv` flow

```bash
source .venv/bin/activate
uv pip install -r requirements.txt
uv run jupyter-book build .
```

Observed results:

- `uv` was available in the environment.
- Dependency installation succeeded.
- `jupyter-book --version` reported `v2.1.2`.
- The build command failed with an `EISDIR` error rather than producing `_build/html/`.

#### `pip` flow

The following flow was tested in a fresh temporary virtual environment:

```bash
pip install -r requirements.txt
jupyter-book --version
jupyter-book build /path/to/cdifbook
```

Observed results:

- `pip` installed successfully from the same `requirements.txt`.
- `jupyter-book --version` again reported `v2.1.2`.
- The build failed with the same `EISDIR` error.

Conclusion:

- The failure is not specific to `uv`.
- Both `pip` and `uv` currently reproduce the same broken setup because they resolve the same incompatible `jupyter-book` version from the unpinned dependency file.

This supports treating the issue as a repository dependency-management problem rather than a package-manager problem.

## Interpretation

The main decision is not whether `pip` or `uv` should be used. The main decision is which Jupyter Book toolchain this repository intends to support.

There are two plausible paths.

### Option A: Keep the current classic Jupyter Book layout

This is the lower-risk option and the fastest path back to a predictable contributor workflow.

Expected actions:

- Pin a compatible `jupyter-book` version in `requirements.txt`.
- Update `_config.yml` repository metadata to match the actual repository layout and branch after confirming whether `path_to_book` is intended for source-link generation.
- Add `.venv/` to `.gitignore`.
- Keep the current README workflow, whether contributors use `pip` or `uv`, adjusted only as needed for the pinned version.

Benefits:

- Minimal content migration.
- Low disruption to contributors.
- Restores a working local build with limited change scope.

### Option B: Migrate intentionally to the newer Jupyter Book / MyST toolchain

This is the larger change and should be treated as a deliberate migration project.

Expected actions:

- Review the new CLI and project structure expectations.
- Update build commands and contributor instructions.
- Validate whether `_config.yml` and `_toc.yml` remain correct as-is or need migration.
- Potentially introduce modern project metadata and locking for a cleaner and reproducible environment workflow.

Benefits:

- Aligns the repo with the current upstream toolchain.
- May support a more modern and explicit build pipeline.

Tradeoff:

- This is a migration project, not just a dependency update.

## Recommended Immediate Next Step

Before making broader changes, decide whether the repository should remain on the classic Jupyter Book workflow or migrate forward.

If the goal is stability and fast recovery, pin the Jupyter Book version and correct the stale metadata.

If the goal is modernization, plan the migration explicitly and update documentation, configuration, and dependencies together rather than incrementally.

## Recommended Resolution

Recommended near-term action: stabilize the current setup unless there is already an active decision to migrate.

That means:

- pin a compatible `jupyter-book` version in `requirements.txt`,
- verify and correct repository metadata in `_config.yml`,
- add `.venv/` to `.gitignore`,
- and keep the documented contributor workflow aligned with the pinned toolchain.

Reasoning:

- This is the smallest change that restores reproducible local testing.
- It avoids turning a dependency break into an unplanned migration project.
- It creates a stable baseline from which a later migration can be planned deliberately, if desired.
